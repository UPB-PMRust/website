Project DescriptionFlash-Point is a hardware-based reaction game built for two players. The idea came from wanting to create a physical competition that removes the "input lag" often found in modern gaming apps. The concept is simple: a central LED turns on after a random delay, and players must race to hit their button first. The system captures the response time in milliseconds and displays the winner's result on an OLED screen.To make the device more professional, I decided to go beyond a simple timer. I integrated a data-logging system that saves match results to an external SD card. This allows players to keep a history of their performances and track high scores even after the power is turned off.Hardware DesignFor this project, I chose the Raspberry Pi Pico (RP2040). It’s an ideal choice because it’s reliable and has a very precise internal clock, which is essential for measuring speed in milliseconds. The hardware is kept minimalist on a breadboard to ensure the connections are stable and easy to troubleshoot during a live demo.Components used:Microcontroller: Raspberry Pi PicoDisplay: SSD1306 OLED (connected via I2C)Storage: Micro SD Card Module (connected via SPI)Input: Two physical push buttons for the playersOutput: A high-brightness LED for the starting signalEssentials: A breadboard, jumper wires, and resistors (220Ω and 10kΩ) to protect the components.Software DesignThe software is written in Rust, which is perfect for this project because it offers great performance without the memory safety risks of older languages. I used an asynchronous approach to make sure the game logic stays fast and responsive.Key technical features:Hardware Interrupts: To be 100% fair, I used GPIO interrupts to detect button presses. This tells the processor to stop exactly when a button is touched, capturing the time with microsecond accuracy.Storage Management: Using the SPI protocol and the embedded-sdmmc crate, the software manages a FAT32 file system on the SD card. It writes match data into a .csv file that can be read later.Debouncing Logic: Buttons can be "noisy" electrically. I implemented a software debouncing algorithm to make sure a single press isn't counted twice.State Machine: The code is organized into a state machine that handles everything from the "Waiting" phase to the "Saving Data" phase.Bill of MaterialsComponentQuantityConnection TypeRaspberry Pi Pico1Main ControllerSSD1306 OLED1I2C ProtocolMicro SD Module1SPI ProtocolPush Buttons2Interrupt-based InputLED1Digital OutputMicro SD Card1FAT32 Formatted Storage
